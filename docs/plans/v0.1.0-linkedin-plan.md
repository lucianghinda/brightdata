## Unit 2: Errors + Result Re-export + Version Hookup

**Goal:** Define the gem's error hierarchy and surface `BrightData::Result` as a re-export of simple-result so callers `require "brightdata"` only.

**Requirements trace:** R6

**Approach:**
```ruby
module BrightData
  class Error < StandardError; end
  class ConfigurationError < Error; end  # missing api_token, bad config
  class ArgumentError < Error; end       # invalid input shape, bad URL, etc.
  class AuthError < Error; end           # 401/403 from API
  class HTTPError < Error               # base for transport errors
    attr_reader :status, :body, :response
    def initialize(message, status:, body:, response: nil)
      super(message); @status = status; @body = body; @response = response
    end
  end
  class RateLimitError < HTTPError; end          # 429
  class ServerError < HTTPError; end             # 5xx
  class ScrapeTimeoutError < Error
    attr_reader :snapshot_id
    def initialize(message, snapshot_id:); super(message); @snapshot_id = snapshot_id; end
  end
  class SnapshotFailedError < Error
    attr_reader :snapshot_id, :details
    def initialize(message, snapshot_id:, details: nil); super(message); @snapshot_id = snapshot_id; @details = details; end
  end
end
```

**Patterns:** Standard Ruby error hierarchy — single root, semantically named subclasses, structured data carried as attr_readers (not stuffed into the message).

**Test scenarios:**
- [ ] Happy path: each class inherits from `BrightData::Error`
- [ ] `HTTPError` retains `status` and `body`
- [ ] `ScrapeTimeoutError` retains `snapshot_id`
- [ ] `BrightData::Result.success(:ok)` returns a success Result (proves re-export works)

**Verification:** `bundle exec rake test` green; YARD-documented every class.

## Unit 3: HTTP Wrapper (`BrightData::HTTP`)

- `test/fixtures/rate_limit.json` — sample 429 body

**Approach:**
```ruby
class BrightData::HTTP
  BASE_URL = "https://api.brightdata.com"
  # Must be strictly greater than Bright Data's /scrape 60s API cap, so a slow
  # /scrape returns its fallback {snapshot_id} body instead of TCP-timing out
  # and losing the snapshot reference. See finding E1 in gauntlet review.
  DEFAULT_TIMEOUT = 90

  def initialize(api_token:, base_url: BASE_URL, open_timeout: 10, read_timeout: DEFAULT_TIMEOUT, logger: nil)
    raise ConfigurationError, "api_token required" if api_token.nil? || api_token.empty?
    @api_token = api_token
    @base_url = base_url
    @open_timeout = open_timeout
    @read_timeout = read_timeout
  end

  # @return [Hash, Array] parsed JSON body
  # @raise [AuthError, RateLimitError, ServerError, HTTPError]
  def post(path:, query: {}, body: nil)
    request(method: :post, path:, query:, body:)
  end

  def get(path:, query: {})
    request(method: :get, path:, query:)
  end

  private
  # builds URI, adds Authorization + Content-Type, encodes JSON,
  # parses response, maps status → error
end
```

**Patterns:** Stdlib-only; uses `URI`, `Net::HTTP`, `JSON.parse(symbolize_names: true)`. Keyword args throughout.

**Test scenarios:**
- [ ] Happy path: POST returns parsed JSON hash (200)
- [ ] Empty body: POST with no body sends `nil` body (used by GET-equivalents)
- [ ] Query params encoded correctly (multiple, with special chars)
- [ ] 401 → `AuthError`
- [ ] 403 → `AuthError`
- [ ] 429 → `RateLimitError` carrying status + body
- [ ] 500 → `ServerError`
- [ ] 502/503/504 → `ServerError`
- [ ] Network timeout → `HTTPError` with status nil
- [ ] Missing api_token at construction → `ConfigurationError`
- [ ] Authorization header is `Bearer <token>` verbatim

**Verification:** All branches mocked via WebMock; line coverage on `http.rb` ≥ 95%.

## Unit 5: Snapshot (`BrightData::Snapshot`)

**Goal:** First-class value object returned by every `.trigger` call. Encapsulates snapshot ID, polling, and result retrieval.

**Requirements trace:** R4, R5, R14

**Dependencies:** U3 (HTTP), U2 (errors, result)

**Files:**
- `lib/brightdata/snapshot.rb` — `BrightData::Snapshot` class
- `lib/brightdata.rb` — wire require
- `test/brightdata/snapshot_test.rb` — wait state machine + result paths
- `test/fixtures/snapshot_progress_running.json`
- `test/fixtures/snapshot_progress_ready.json`
- `test/fixtures/snapshot_progress_failed.json`
- `test/fixtures/snapshot_results_sample.json`

**Approach:**
```ruby
class BrightData::Snapshot
  STATUSES = %i[starting running ready failed].freeze
  attr_reader :id, :status, :http, :result_parser

  # @param id [String] snapshot_id from /datasets/v3/trigger
  # @param http [BrightData::HTTP]
  # @param result_parser [#call] callable mapping raw JSON array → typed Data instances
  def initialize(id:, http:, result_parser: ->(raw) { raw })
    @id = id; @http = http; @result_parser = result_parser; @status = :starting
  end

  # GET /datasets/v3/progress/{id}
  # @return [Symbol] one of STATUSES
  def progress
    payload = @http.get(path: "/datasets/v3/progress/#{@id}")
    @status = payload.fetch(:status).to_sym
  end

  def ready?  = @status == :ready
  def failed? = @status == :failed

  # Polls until terminal status or timeout.
  # @param timeout [Numeric] seconds; default 300
  # @param poll_interval [Numeric] seconds; default 5
  # @return [BrightData::Result::Success<Array>] on :ready (results array, parsed)
  # @return [BrightData::Result::Failure<Hash>] on :failed (details payload)
  # @raise [BrightData::ScrapeTimeoutError] if timeout elapses before terminal
  def wait(timeout: 300, poll_interval: 5)
    deadline = monotonic_now + timeout
    loop do
      progress
      return success_with_results if ready?
      return failure_with_details if failed?
      raise ScrapeTimeoutError.new("snapshot still #{@status} after #{timeout}s", snapshot_id: @id) if monotonic_now >= deadline
      sleep(poll_interval)
    end
  end

  # GET /datasets/v3/snapshot/{id}?format=json
  # @return [Array] raw response array (also accessible via wait's Success)
  def results
    @http.get(path: "/datasets/v3/snapshot/#{@id}", query: { format: "json" })
  end

  private
  def success_with_results
    raw = results
    Result.success(@result_parser.call(raw))
  end
  def failure_with_details
    payload = @http.get(path: "/datasets/v3/progress/#{@id}")
    Result.failure(payload)
  end
  def monotonic_now = Process.clock_gettime(Process::CLOCK_MONOTONIC)
end
```

**Patterns:** Single-class state machine; sleep-based polling with monotonic clock; injected parser via `Proc` (enables endpoint classes to transform raw arrays into typed `Data` instances without subclassing Snapshot).

**Test scenarios:**
- [ ] Happy path: progress → running → ready → wait returns Success(parsed array)
- [ ] Failed path: progress → failed → wait returns Failure(details)
- [ ] Timeout path: progress stays :running → wait raises ScrapeTimeoutError with snapshot_id
- [ ] poll_interval is honored (mock `sleep` and assert called with correct argument)
- [ ] Default parser passes raw through unchanged
- [ ] Custom parser is invoked with raw array on success
- [ ] `Process::CLOCK_MONOTONIC` used (test by stubbing & ensuring no `Time.now` calls)
- [ ] Unknown status string from API → raises BrightData::Error (defensive)

**Verification:** Snapshot lifecycle covered by 8+ tests; all status branches exercised; ScrapeTimeoutError carries snapshot_id.

**Planning-time unknowns:**
- Exact path for snapshot results: `/datasets/v3/snapshot/{id}` is the documented path; confirm during fixture capture. **Deferred to Planning.**

## Unit 6: Client + LinkedIn Namespace + Endpoint Mixin

**Goal:** `BrightData::Client.new(api_token:)` is the public entry point; `client.linkedin` returns a `BrightData::LinkedIn::Namespace` with five sub-namespace accessors; a shared `Endpoint` mixin provides reusable `trigger_*`/`scrape_*` helpers for endpoint classes to compose.

**Requirements trace:** R1, R2, R3, R6, R16, R19

**Dependencies:** U3, U4, U5

**Files:**
- `lib/brightdata/client.rb`
- `lib/brightdata/linkedin/namespace.rb`
- `lib/brightdata/linkedin/endpoint.rb` — mixin module
- `lib/brightdata.rb` — wire requires
- `test/brightdata/client_test.rb`
- `test/brightdata/linkedin/endpoint_test.rb` — exercises mixin via a fake endpoint class
- `test/fixtures/linkedin/trigger_response.json` — `{"snapshot_id": "..."}`

**Approach:**
```ruby
class BrightData::Client
  attr_reader :http
  def initialize(api_token:, base_url: BrightData::HTTP::BASE_URL)
    @http = BrightData::HTTP.new(api_token:, base_url:)
  end
  def linkedin
    @linkedin ||= BrightData::LinkedIn::Namespace.new(http: @http)
  end
end

class BrightData::LinkedIn::Namespace
  def initialize(http:); @http = http; end
  def profiles  = @profiles  ||= BrightData::LinkedIn::Profiles.new(http: @http)
  def companies = @companies ||= BrightData::LinkedIn::Companies.new(http: @http)
  def jobs      = @jobs      ||= BrightData::LinkedIn::Jobs.new(http: @http)
  def posts     = @posts     ||= BrightData::LinkedIn::Posts.new(http: @http)
  def people    = @people    ||= BrightData::LinkedIn::People.new(http: @http)
end

module BrightData::LinkedIn::Endpoint
  # @api private
  # Subclasses set DATASET_KEY constant; mixin reads BrightData::Datasets.id_for(KEY)
  # Subclasses set EXTRA_QUERY hash for type=discover_new etc.
  # Subclasses set RESULT_TYPE class (Data class) and INPUT_TYPE class (Data class)
  TRIGGER_PATH = "/datasets/v3/trigger"
  SCRAPE_PATH  = "/datasets/v3/scrape"
  SYNC_TIMEOUT_SECONDS = 60

  private

  def trigger_with(dataset_key:, inputs:, extra_query: {})
    payload = @http.post(
      path: TRIGGER_PATH,
      query: { dataset_id: BrightData::Datasets.id_for(dataset_key) }.merge(extra_query),
      body: { input: inputs.map(&:to_h) }
    )
    BrightData::Snapshot.new(
      id: payload.fetch(:snapshot_id),
      http: @http,
      result_parser: result_parser_for(dataset_key)
    )
  end

  def scrape_with(dataset_key:, inputs:, extra_query: {})
    payload = @http.post(
      path: SCRAPE_PATH,
      query: { dataset_id: BrightData::Datasets.id_for(dataset_key) }.merge(extra_query),
      body: { input: inputs.map(&:to_h) }
    )
    # /scrape may return an array (sync success) OR { snapshot_id: ... } if > 60s
    if payload.is_a?(Hash) && payload[:snapshot_id]
      raise BrightData::ScrapeTimeoutError.new(
        "Bright Data /scrape exceeded 60s — switch to .trigger + Snapshot#wait",
        snapshot_id: payload[:snapshot_id]
      )
    end
    result_parser_for(dataset_key).call(payload)
  end

  # Override in endpoint class to map raw → typed Data array
  def result_parser_for(_dataset_key); ->(raw) { raw }; end
end
```

**Patterns:** Mixin (not inheritance) for endpoint composition; memoized sub-namespace accessors; keyword-only public API; positional internal helpers ok.

**Test scenarios:**
- [ ] `Client.new(api_token: "x")` succeeds; `Client.new` (no kwarg) raises ArgumentError
- [ ] `Client.new(api_token: nil)` → ConfigurationError (from HTTP)
- [ ] `client.linkedin` returns memoized Namespace (same object on repeat call)
- [ ] `client.linkedin.profiles` etc. return memoized endpoint classes
- [ ] Fake endpoint using the mixin: `.trigger` returns Snapshot with parsed `snapshot_id`
- [ ] Fake endpoint using the mixin: `.scrape` returns parsed array on 200
- [ ] Fake endpoint using the mixin: `.scrape` raises `ScrapeTimeoutError` when response carries `snapshot_id`
- [ ] `trigger_with` posts to `/datasets/v3/trigger` with correct query and body shape
- [ ] `scrape_with` posts to `/datasets/v3/scrape` with correct query and body shape
- [ ] Auth header is set on every request (verified once at HTTP layer; integration test confirms)

**Verification:** Client + Namespace + Endpoint mixin covered; integration test (mocked) calls `client.linkedin.profiles.trigger(...)` and verifies the request shape end-to-end.

  # @param urls [Array<String>]

  def trigger(urls:)

## Unit 9: LinkedIn Jobs — Scaffolding + Collect by URL

**Goal:** `BrightData::LinkedIn::Jobs` class with `collect_by_url` mode. First of three modes.

**Requirements trace:** R9

**Dependencies:** U8

**Files:**
- `lib/brightdata/linkedin/jobs.rb` — class with `collect_by_url.trigger` and `collect_by_url.scrape` accessors via memoized sub-mode object (see Approach)
- `lib/brightdata/linkedin/types/job.rb` — output type
- `lib/brightdata/linkedin/types/job_url_input.rb` — input type for url mode
- `lib/brightdata.rb` — wire requires
- `test/brightdata/linkedin/jobs_collect_by_url_test.rb`
- `test/fixtures/linkedin/jobs_collect_by_url_response.json`

**Approach:**

The Jobs class has three modes (`collect_by_url`, `discover_by_url`, `discover_by_keyword`) that share infrastructure but differ in dataset query params and input shapes. Two design options:

**Option A (chosen): sub-mode accessor pattern.**

```ruby
class BrightData::LinkedIn::Jobs
  def initialize(http:); @http = http; end

  # @return [BrightData::LinkedIn::Jobs::CollectByUrl]
  def collect_by_url
    @collect_by_url ||= CollectByUrl.new(http: @http)
  end

  # @return [BrightData::LinkedIn::Jobs::DiscoverByUrl]
  def discover_by_url
    @discover_by_url ||= DiscoverByUrl.new(http: @http)
  end

  # @return [BrightData::LinkedIn::Jobs::DiscoverByKeyword]
  def discover_by_keyword
    @discover_by_keyword ||= DiscoverByKeyword.new(http: @http)
  end

  class CollectByUrl
    include BrightData::LinkedIn::Endpoint
    def initialize(http:); @http = http; end
    def trigger(urls:); inputs = urls.map { |u| Types::JobUrlInput.new(url: u) }; trigger_with(dataset_key: :jobs_collect_by_url, inputs:); end
    def scrape(urls:);  inputs = urls.map { |u| Types::JobUrlInput.new(url: u) }; scrape_with(dataset_key:  :jobs_collect_by_url, inputs:); end
    private
    def result_parser_for(_); ->(raw) { raw.map { |h| Types::Job.from_api(h) } }; end
  end
end
```

Usage: `client.linkedin.jobs.collect_by_url.scrape(urls: [...])`.

This is verbose but reads exactly like the Bright Data docs taxonomy and avoids method overload on a single `Jobs` class. Alternative (B): single class with `collect_by_url(urls:)` and `discover_by_url(urls:)` as plain methods, each returning Snapshot — rejected because the "two verbs" (`trigger`/`scrape`) needs to live somewhere, and stacking 6 methods on Jobs (`trigger_collect_by_url`, `scrape_collect_by_url`, …) is uglier than `jobs.collect_by_url.trigger`.

**Patterns:** Each mode is its own class with the Endpoint mixin. Sub-mode classes namespaced under their parent (`Jobs::CollectByUrl`).

**Test scenarios:**
- [ ] `client.linkedin.jobs.collect_by_url.trigger(urls: [...])` returns Snapshot
- [ ] `.scrape(urls: [...])` returns `[Job]`
- [ ] Sub-mode is memoized on parent class
- [ ] `raw:` populated

**Verification:** Collect-by-URL mode of Jobs end-to-end via WebMock.

**Planning-time unknowns:** Confirm `dataset_id` is the same across all three Jobs modes (docs suggest yes — same `gd_lpfll7v5hcqtkxl6l` with `type`/`discover_by` query params switching mode). **Deferred to Planning.**

## Unit 10: LinkedIn Jobs — Discover by URL + Discover by Keyword

**Goal:** Complete the Jobs resource with the two discover modes.

**Requirements trace:** R9

**Dependencies:** U9

**Files:**

- `lib/brightdata/linkedin/types/job_keyword_input.rb` — Data class with all keyword-mode input fields
- `lib/brightdata.rb` — wire requires (if needed)

- `test/fixtures/linkedin/jobs_discover_by_url_response.json`
- `test/fixtures/linkedin/jobs_discover_by_keyword_response.json`

**Approach:**

`DiscoverByUrl` and `DiscoverByKeyword` each include `Endpoint`. The key wrinkle is `extra_query` — the Bright Data docs say discover modes require `type=discover_new` (and `discover_by=keyword` for keyword mode).

```ruby
class BrightData::LinkedIn::Jobs::DiscoverByUrl
  include BrightData::LinkedIn::Endpoint
  EXTRA_QUERY = { type: "discover_new", discover_by: "url" }.freeze
  def initialize(http:); @http = http; end
  def trigger(urls:)
    inputs = urls.map { |u| Types::JobUrlInput.new(url: u) }
    trigger_with(dataset_key: :jobs_discover_by_url, inputs:, extra_query: EXTRA_QUERY)
  end
  def scrape(urls:)
    inputs = urls.map { |u| Types::JobUrlInput.new(url: u) }
    scrape_with(dataset_key: :jobs_discover_by_url, inputs:, extra_query: EXTRA_QUERY)
  end
  private
  def result_parser_for(_); ->(raw) { raw.map { |h| Types::Job.from_api(h) } }; end
end

# Types::JobKeywordInput captures the full input shape from docs
BrightData::LinkedIn::Types::JobKeywordInput = Data.define(
  :location, :keyword, :country, :time_range, :job_type,
  :experience_level, :remote, :company, :selective_search,
  :jobs_to_not_include, :location_radius
)

class BrightData::LinkedIn::Jobs::DiscoverByKeyword
  include BrightData::LinkedIn::Endpoint
  EXTRA_QUERY = { type: "discover_new", discover_by: "keyword" }.freeze
  def initialize(http:); @http = http; end

  # @param queries [Array<BrightData::LinkedIn::Types::JobKeywordInput>]
  # OR accept individual keyword args via convenience? Defer to single shape: callers build inputs.
  def trigger(queries:)
    trigger_with(dataset_key: :jobs_discover_by_keyword, inputs: queries, extra_query: EXTRA_QUERY)
  end
  def scrape(queries:)
    scrape_with(dataset_key: :jobs_discover_by_keyword, inputs: queries, extra_query: EXTRA_QUERY)
  end
  private
  def result_parser_for(_); ->(raw) { raw.map { |h| Types::Job.from_api(h) } }; end
end
```

**Patterns:** `EXTRA_QUERY` constant per mode; for the multi-field keyword input, the caller constructs `JobKeywordInput` instances rather than passing 11 keyword args directly. This keeps the endpoint API stable and pushes input validation into one place.

**Test scenarios:**
- [ ] `jobs.discover_by_url.scrape(urls: [...])` → `[Job]`
- [ ] `jobs.discover_by_keyword.scrape(queries: [JobKeywordInput.new(location: "NYC", keyword: "ruby", ...)])` → `[Job]`
- [ ] Trigger versions return Snapshot
- [ ] Request URL has correct `type=discover_new` and `discover_by` query params
- [ ] Missing required fields in `JobKeywordInput` (e.g., `location:`) → ArgumentError from Data.new (Ruby native)

**Verification:** Both discover modes work end-to-end via WebMock; query params verified verbatim.

**Planning-time unknowns:**
- Does `JobKeywordInput` need all 11 fields as required by `Data.define`, or are some optional? Docs say `location` is required; others optional. **Decision: define with all fields; caller passes `nil` for ones they skip; we serialize `to_h` and strip nil values before posting.** Add `.to_api_hash` helper on input types that omits nils. **Deferred to Planning** — update `Endpoint#trigger_with` to call `.to_api_hash` if responded to, else `.to_h`.
