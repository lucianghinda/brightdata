# v0.1.0 Execution Prompt (for codex `/goal` or any LLM)

Paste the prompt block below into codex via `/goal`, or hand it to any LLM agent that
can read files and run shell commands in this repo. Substitute nothing — the wording is
load-bearing.

---

```
/goal Implement the `brightdata` Ruby gem v0.1.0 by executing all numbered Tasks defined in `docs/plans/v0.1.0-implementation-tasks.md`.

Before doing anything else, read the following files in full and treat their guidance as load-bearing for every Task you implement:

1. `docs/plans/v0.1.0-implementation-tasks.md` — the task list and code skeletons. This is your primary spec.
2. `docs/product/mission.md` — why the gem exists.
3. `docs/product/mvp-plan.md` — what ships in v0.1.0 and what is explicitly out of scope.
4. `docs/product/tech-stack.md` — language version, dependencies, project layout, decision principles.
5. `docs/plans/v0.1.0-linkedin-plan.md` — architectural Units, requirements trace (R1–R20), and Phase 5 gauntlet amendments.

If any of those files is missing, stop and ask before continuing.

Tasks are numbered 1 through 23 inside `v0.1.0-implementation-tasks.md`. Treat the number as the execution order. Work through them one by one, in ascending order. Do not skip ahead, do not batch tasks, do not reorder unless the task's own dependency section explicitly requires it.

For each Task:
1. Read the full Task section (Goal, Requirements, Files, Steps, code skeletons, Verification, Commit message).
2. Implement only what that Task specifies. Do not pull work forward from later Tasks.
3. Use the provided code skeletons as the starting point. Adjust only where the skeleton says "confirm at fixture capture" or where a documented planning-time unknown applies.
4. Keep the diff small, scoped, and consistent with the patterns established in earlier Tasks.
5. Run the Task's Verification section before considering it done. At minimum that means:
   - `bundle exec rake test` exits 0
   - `bundle exec standardrb` exits 0
   - `bundle exec yard --fail-on-warning` exits 0 (after Task 20; before that, yard may have legitimate warnings)
6. Create exactly one git commit per Task, using the Commit message string from the Task header verbatim. Reference the Task number in the commit body if you want to add context.
7. Start the next Task only after the current Task's commit has landed and verification is green.

Hard rules — non-negotiable, override any quality-bar tradeoff:

- Ruby version target is `>= 3.2.0`. Use `Data.define`, pattern matching, and endless methods where they improve clarity. Do not add polyfills for older Rubies.
- Public API uses keyword arguments only. No positional public method parameters, ever. Internal helpers may use positional.
- Every `.rb` file starts with `# frozen_string_literal: true`.
- Runtime dependencies are limited to `simple-result` only. Adding any other runtime gem requires updating `docs/product/tech-stack.md` first and asking before proceeding.
- All HTTP calls go through `BrightData::HTTP` (Task 7). No `Net::HTTP`, `URI.open`, `Faraday`, or similar called from anywhere else in the codebase.
- Errors raised by the gem MUST come from the `BrightData::Errors` hierarchy (Task 5). Never raise stdlib `::ArgumentError` from user-facing code — raise `BrightData::ArgumentError`.
- `BrightData::HTTP::DEFAULT_TIMEOUT` is 90 seconds, not 60. This is intentional (Phase 5 finding E1) and prevents silent snapshot loss against Bright Data's 60s `/scrape` cap. Do not "fix" it.
- Every output `Data.define` class has a `:raw` field carrying the entire parsed-hash response. Document this in YARD with `@note Use #raw to access fields not yet typed by this gem.`
- Every public method, class, module, and parameter has a YARD comment block (`@param`, `@return`, `@raise`, `@example` where helpful). CI runs `yard --fail-on-warning` from Task 20 onward and must stay green.
- WebMock is enabled in `test_helper.rb` with `WebMock.disable_net_connect!(allow_localhost: true)`. No test makes a real network call. Live tests live under `test/live/` and are gated on `ENV["BRIGHTDATA_LIVE"]`.
- Anything listed under "Out of Scope for v0.1.0" in `v0.1.0-implementation-tasks.md` §5 is forbidden in this build. If you find yourself implementing webhooks, retries, Rails integration, RBS, CLI, concurrent batches, or other namespaces — stop and ask.

Quality bar — every change must optimize for, in this order:

- Simplicity: smallest diff that satisfies the Task; no speculative abstractions; YAGNI. The code skeletons in the Task file are the target, not a starting point to embellish.
- Maintainability: clear seams between Client/HTTP/Snapshot/Endpoint/Datasets/Types; no hidden coupling between endpoint classes; no dead code; no leftover scaffolding.
- Readability: intent-revealing names matching the Bright Data docs taxonomy (`collect_by_url`, `discover_by_keyword`, etc.); short methods; explicit control flow.
- Easy to debug: explicit error messages that name the offending input and the expected shape; no silent `rescue`; named locals over inline chains where it helps a reader.
- Ruby idiomatic: `Data.define` for value objects; keyword args; frozen string literals; `Comparable`/`Enumerable` over manual loops; no Java-flavoured Ruby; pattern matching where it reads better than conditionals.
- Test discipline: WebMock fixtures captured from real responses (not invented); one logical assertion concept per test method; happy path + edge cases (nil, empty, malformed, auth, rate-limit, 5xx) covered per the Test Scenarios section of each Task.

Fixture capture: When a Task requires a JSON fixture from the real Bright Data API (Tasks 11, 12, 13, 14, 15, 16, 17), follow the fixture capture procedure documented under Task 11. Sanitize any personal data before committing the fixture. If you cannot reach the real API, generate a fixture that matches the documented response shape from the Bright Data docs URL referenced in the Task, mark it with a `// SYNTHETIC` comment in an adjacent README note, and call this out in your progress report so the human can replace it with real captured data before v0.1.0 ships.

Placeholder dataset IDs: 8 entries in `BrightData::Datasets::LINKEDIN` start as placeholder values. Task 18 resolves them with real values captured during fixture work. Do NOT ship v0.1.0 (Task 22 build, Task 23 release) while any placeholder remains. CI for Task 22 must grep for placeholder markers and fail if found.

After completing each Task, output one line of progress in this exact form so a human watching the log can follow along:

```
✓ Task <N> — <commit subject> — tests: <pass/fail count>, lint: <pass/fail>, docs: <pass/fail>
```

Stop and ask only if:
- A required file (planning doc) is missing or unreadable.
- A Task's instructions conflict with a hard rule or with an earlier Task's commit.
- The Bright Data API returns a response shape that materially diverges from the docs and the typed output `Data` class needs new fields beyond `:raw`.
- A real-world blocker prevents verification (e.g., no Bright Data token available for live tests required by Task 18).
- A destructive action would be required (force push, history rewrite, branch deletion).

Otherwise, work through all 23 Tasks end to end. The session is complete when:
- All 23 Tasks have a corresponding commit on the current branch.
- `bundle exec rake test && bundle exec standardrb && bundle exec yard --fail-on-warning` all exit 0.
- `BRIGHTDATA_LIVE=1 BRIGHTDATA_API_TOKEN=... bundle exec rake test:live` all exit 0 (gated on token availability — if no token, report this and skip Task 18+23 manual steps).
- `git grep` for placeholder dataset markers returns nothing in `lib/`.
- The Acceptance Criteria checklist in `v0.1.0-implementation-tasks.md` §4 is fully satisfied.
```
